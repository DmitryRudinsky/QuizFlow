# База данных, миграции и JPA Entities

Здесь описано, как устроена база данных в этом проекте: что хранится, как создаётся схема, и как Kotlin-код связан с таблицами.

---

## Как вообще всё работает

Приложение подключается к PostgreSQL. Когда оно запускается, сначала отрабатывает Flyway — он смотрит, какие миграции уже применялись, и выполняет те, что ещё не были. Только после этого Spring Boot поднимает остальное. Hibernate при старте проверяет, что таблицы в базе совпадают с тем, что описано в Kotlin-классах. Если что-то не совпадает — приложение не запустится с ошибкой.

Цепочка такая: `application.yml` → `HikariCP` (пул соединений) → `JDBC Driver` (PostgreSQL) → сама база.

---

## Конфигурация подключения

Всё, что касается подключения, лежит в `server/src/main/resources/application.yml`. Пароль и другие секреты туда не вписываются напрямую — они берутся из переменных окружения, которые в свою очередь загружаются из файла `.env` в корне репозитория (сам `.env` в git не коммитится, есть только `.env.example` как шаблон).

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/${POSTGRES_DB:quizflow}
    username: ${POSTGRES_USER:quizflow}
    password: ${POSTGRES_PASSWORD}
```

Запись `${POSTGRES_DB:quizflow}` означает: возьми переменную окружения `POSTGRES_DB`, а если она не задана — используй `quizflow` как значение по умолчанию. Для пароля значения по умолчанию нет — специально, чтобы не запустить приложение с пустым паролем случайно.

`ddl-auto: validate` — Hibernate при старте только сверяет схему с entities, но сам ничего в базе не создаёт и не меняет. Всё это делает Flyway.

## Flyway и миграции

Flyway — это инструмент для управления изменениями в базе данных. Идея простая: каждое изменение схемы описывается в отдельном SQL-файле с версией. Flyway следит за тем, какие файлы уже применялись (хранит это в таблице `flyway_schema_history`), и применяет только новые.

Файлы миграций лежат в `server/src/main/resources/db/migration/`.

Формат имени файла: `VYYYY.MM.DD.HH.MM.SS__описание.sql`. Дата и время нужны, чтобы файлы выполнялись в правильном порядке — Flyway сортирует их по версии.

В проекте четыре миграции:

`V2026.04.12.10.00.00__create_enums.sql` — создаёт PostgreSQL-типы (ENUM). Это специальные типы, которые ограничивают набор допустимых значений прямо на уровне базы. Если попробовать записать значение, которого нет в ENUM — база выдаст ошибку.

`V2026.04.12.10.00.01__create_tables.sql` — создаёт все восемь таблиц. Здесь описаны колонки, типы данных, внешние ключи и ограничения.

`V2026.04.12.10.00.02__create_indexes.sql` — добавляет индексы. Индекс — это отдельная структура данных внутри базы, которая позволяет быстро найти строки по конкретной колонке. Без индекса база перебирает все строки подряд; с индексом — идёт напрямую к нужным.

`V2026.04.12.10.00.03__add_unique_constraints.sql` — добавляет дополнительные уникальные ограничения, которые неудобно было записать в `CREATE TABLE`. В частности, частичный уникальный индекс на `session_participants` — один зарегистрированный пользователь не может быть в одной сессии дважды, а гости (у которых `user_id IS NULL`) этим ограничением не затронуты.

## Схема базы данных

### Таблицы

**users** — пользователи системы. Каждый пользователь может быть организатором (создаёт квизы и запускает сессии) или участником. Пароль хранится в виде хеша — никогда в открытом виде.

**quizzes** — квизы. У каждого квиза есть владелец (`created_by` — ссылка на пользователя). Квиз содержит вопросы и настройки, но сами они в отдельных таблицах.

**quiz_settings** — настройки квиза: время на вопрос, режим подсчёта очков, нужно ли перемешивать вопросы и так далее. Это таблица 1-к-1 с `quizzes`: у каждого квиза ровно одни настройки. Первичный ключ `quiz_id` одновременно является внешним ключом к `quizzes` — то есть отдельного `id` у настроек нет, они идентифицируются по квизу.

**questions** — вопросы. Каждый вопрос принадлежит одному квизу. Поле `position` определяет порядок вопросов внутри квиза (0, 1, 2...). Пара `(quiz_id, position)` уникальна — нельзя иметь два вопроса с одинаковой позицией в одном квизе.

**answers** — варианты ответов. Каждый ответ принадлежит одному вопросу. Флаг `is_correct` обозначает правильный ответ; при этом участникам этот флаг не отправляется — только организатору. Пара `(question_id, text)` уникальна — нельзя добавить два одинаковых варианта к одному вопросу.

**sessions** — живые сессии квизов. Когда организатор запускает квиз в реальном времени, создаётся запись в этой таблице. `room_code` — короткий код вроде "QUIZ42", по которому участники находят комнату. `current_question_index` меняется по ходу сессии, когда организатор переходит к следующему вопросу.

**session_participants** — участники конкретной сессии. `user_id` может быть NULL — это значит, что участник зашёл как гость без аккаунта. `score` накапливается по ходу сессии. Зарегистрированный пользователь не может войти в одну сессию дважды (частичный уникальный индекс на `session_id, user_id WHERE user_id IS NOT NULL`).

**participant_answers** — ответы участников на вопросы. `answer_ids` — массив UUID, потому что вопрос может допускать несколько правильных вариантов. Для вопроса с одним ответом это массив из одного элемента. Пара `(participant_id, question_id)` уникальна — участник не может ответить на один вопрос дважды.

### ENUM-типы

Вместо того чтобы хранить строки произвольного содержания, в базе используются типизированные перечисления:

- `user_role`: `organizer`, `participant`
- `question_type`: `text`, `image` (вопрос с текстом или с картинкой)
- `answer_type`: `single`, `multiple` (один правильный ответ или несколько)
- `scoring_mode`: `standard`, `streak`, `time_bonus` (обычные очки, бонус за серию, бонус за скорость)
- `show_answers`: `after_each`, `end_only` (показывать правильные ответы после каждого вопроса или только в конце)
- `session_status`: `waiting`, `active`, `paused`, `ended`

## JPA Entities

JPA (Jakarta Persistence API) — это стандартный способ в Java/Kotlin работать с реляционными базами через объекты. Вместо того чтобы вручную писать SQL-запросы для каждой операции, ты описываешь классы, которые Hibernate (реализация JPA) автоматически превращает в запросы.

Все entity-классы лежат в `server/src/main/kotlin/com/quizflow/domain/`.

### Основные аннотации

`@Entity` — говорит, что этот класс представляет строку в таблице.

`@Table(name = "...")` — указывает имя таблицы. Без этой аннотации Hibernate взял бы имя класса, но лучше указывать явно.

`@Id` и `@GeneratedValue(strategy = GenerationType.UUID)` — обозначают первичный ключ. `GenerationType.UUID` означает, что идентификатор генерируется как случайный UUID. Это делает и база (`gen_random_uuid()`), и Kotlin-код (`UUID.randomUUID()`) — они оба правильны, просто в разных контекстах.

`@Column` — описывает колонку. Можно указать имя (`name`), допустимость NULL (`nullable`), длину строки (`length`), а также `columnDefinition` для нестандартных типов вроде PostgreSQL ENUM или `uuid[]`.

`@Enumerated(EnumType.STRING)` — говорит Hibernate хранить enum как строку, а не как число. Это важно: если хранить числом, то после добавления нового значения в enum порядок поломается.

`@Column(updatable = false)` — Hibernate никогда не включит эту колонку в UPDATE-запросы. Так устроены поля `created_at` — они пишутся один раз при создании и больше не меняются.

### Связи между таблицами

`@ManyToOne` + `@JoinColumn` — связь "многие к одному". `@JoinColumn(name = "quiz_id")` говорит, что в текущей таблице есть колонка `quiz_id`, которая хранит внешний ключ. Эта сторона связи "владеет" внешним ключом — то есть именно здесь в базе лежит колонка со ссылкой.

`@OneToMany(mappedBy = "quiz")` — обратная сторона той же связи. `mappedBy = "quiz"` означает: "ключ лежит не здесь, а в поле `quiz` у связанного класса". Эта аннотация нужна только для удобного доступа из кода — в базе никакой дополнительной колонки не создаётся.

`@OneToOne` + `@MapsId` — специальный случай для `QuizSettings`. Здесь `quiz_id` одновременно является и первичным ключом настроек, и внешним ключом на квиз. `@MapsId` говорит Hibernate: возьми ID из связанного объекта (`quiz.id`) и используй его как ID этого объекта.

`FetchType.LAZY` — связанный объект загружается из базы только тогда, когда к нему реально обращаются в коде, а не сразу вместе с основным объектом. Это стандартная практика для оптимизации — без LAZY каждый запрос тянул бы за собой всё дерево связей.

`CascadeType.ALL` — операции на родительском объекте (сохранение, удаление) автоматически применяются к дочерним. Например, удаление квиза удаляет и все его вопросы.

`orphanRemoval = true` — если элемент убрать из коллекции в памяти, Hibernate при следующем сохранении удалит его и из базы. Без этого удаление из списка в коде не повлияло бы на базу.

### Специальный тип: массив UUID

В `ParticipantAnswer` поле `answerIds` имеет тип `Array<UUID>` и хранится как PostgreSQL-массив (`uuid[]`). Стандартный JPA не умеет работать с массивами PostgreSQL, поэтому используется аннотация из Hibernate:

```kotlin
@JdbcTypeCode(SqlTypes.ARRAY)
@Column(columnDefinition = "uuid[]")
val answerIds: Array<UUID>
```

`@JdbcTypeCode(SqlTypes.ARRAY)` указывает Hibernate, какой именно JDBC-тип использовать при чтении и записи.

### Kotlin enum и PostgreSQL ENUM

Enum-классы в Kotlin объявлены с именами в нижнем регистре (`organizer`, `participant`), потому что PostgreSQL ENUM создавался с точно такими же значениями. Hibernate при `EnumType.STRING` записывает именно `.name` у enum — строку как она называется в коде. Поэтому регистр должен совпадать.

## Что проверить если что-то сломалось

Посмотреть, какие миграции применились:
```sql
SELECT * FROM flyway_schema_history ORDER BY installed_rank;
```

Посмотреть список таблиц:
```sql
\dt
```

Посмотреть структуру конкретной таблицы:
```sql
\d session_participants
```

Если приложение не запускается с ошибкой про схему — скорее всего расхождение между тем, что Hibernate видит в entity, и тем, что реально есть в базе. Надо смотреть stacktrace: там написано, какая именно колонка или таблица не найдена.
